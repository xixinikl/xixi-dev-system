# 我的开发系统：仓库地图

平时只需要记住一个名字：`xixi-dev-system`。其他仓库各司其职，不需要手动来回调用。

| 仓库 | 它负责什么 | 平时需要你打开吗 |
|---|---|---|
| `xixi-dev-system` | 总入口。接入项目、检查环境、启动预览、收集证据、安排验收。 | 偶尔；新项目第一次接入时使用。 |
| `xixi-agent-profile` | 保存跨项目都成立的个人偏好和经过验证的经验。 | 通常不用；系统读取它。 |
| `standard-project-workflow` | 规定一个项目从需求到交付必须经过哪些步骤和证据门。 | 通常不用；正式开发任务自动遵守。 |
| `codex-acceptance-factory` | 执行每日检查、保守修复和验收取证。 | 出现失败证据时再看。 |
| `quality-hub` | 给人看的首页：本周是否需要处理、唯一下一步、最新周报。 | 需要看状态时打开。 |

## 它们怎么配合

1. 新项目运行一次 `xixi-dev-system onboard`，仓库里会生成一个很小的项目适配文件。
2. Codex 从个人偏好和项目适配文件理解“你喜欢怎样工作”以及“这个项目怎样运行”。
3. 标准流程约束开发过程，验收工厂产出可核对的结果。
4. 每周回顾只把有证据、重复出现或影响重大的经验写回个人系统。
5. Quality Hub 只展示结论和下一步，不展示机器日志。

这些仓库可以分开存在。分开能保留边界、历史和安装方式；它们通过稳定的文件格式与 GitHub 事实协作，不靠复制彼此的代码。

## 新项目最短用法

在项目根目录运行：

```bash
xixi-dev-system onboard
xixi-dev-system doctor --project .
```

系统会自动识别常见的 Node、静态网页项目。识别不了启动方式时，它会明确告诉你只需补哪一项，不会假装配置成功。

## 视觉一致性的边界

一致性只要求同一个产品内部统一。不同项目默认独立选择视觉方向；Quality Hub、官网、学习产品或游戏界面不应互相套用配色和版式，除非你明确指定。
