# Xixi 实时开发预览 Goal

> **唯一权威入口**：本文是“项目预览中心实时更新”工作的唯一事实合同。聊天中的描述不能替代本文、可执行 Goal 状态、测试或浏览器证据。

## 当前事实

- 项目中心已经能注册公途、思维风暴和码上冒险，显示当前/近期分支，并在独立端口启动预览。
- 当前检出的分支直接从真实工作目录运行，因此修改文件后刷新可以看到新内容；历史分支会使用独立 detached worktree。
- 思维风暴当前使用普通 Node 服务，没有浏览器自动刷新；码上冒险前端有 Vite HMR，但 API 端口固定为 `4317`；公途前端和后端仍有 `8089/8888` 固定端口历史。
- 三个来源工作副本都有用户或其他 Agent 的未提交内容，不能清理、覆盖或混入本 Goal 的提交。
- 项目中心基础能力在 `xixi-dev-system` PR #13；本 Goal 的系统改动必须以该分支为基线单独交付。

## 目标

让用户在项目中心点击当前开发分支后获得真正可持续的实时预览：前端修改自动呈现，服务端修改按项目能力自动重启，分支端口和数据边界清晰可见；对于暂时无法完成的旧后端隔离，留下足够证据让下一轮可以直接修复。

## 范围

- 项目中心区分“实时开发分支”和“快照分支”。
- Xixi 适配器支持一个公开网页端口和额外服务端口，并把端口、命名空间、worktree、日志写入运行状态。
- 思维风暴当前开发分支具备自动刷新；服务端代码具备 watch/restart。
- 码上冒险 Vite 与 API 使用动态端口，数据库使用分支命名空间路径。
- 公途前端使用动态端口并自动刷新；对后端动态端口、同源代理和数据库隔离做真实实现尝试。
- 桌面与手机页面验收、文件改动触发证据、项目测试、系统测试和分仓 PR。

### 不包含

- 不自动合并任何 PR，不把未审阅的 stacked PR 直接合入 `main`。
- 不把真实用户数据库复制到分支预览；预览数据默认隔离或明确标记共享/仅前端。
- 不修改或提交三个原始脏工作副本中的既有用户改动。
- 不为历史远端分支承诺“实时”；未被编辑器检出的 detached worktree 明确称为“快照预览”。
- 不用轮询截图伪装 HMR；自动更新必须由浏览器 URL/DOM 变化、重载计数、服务日志或框架事件证明。

## 交互合同

- 当前检出的工作目录：按钮为“实时预览”，状态为“实时更新”。
- 已在其他普通 worktree 检出的分支：按钮仍为“实时预览”，因为对该目录的编辑会直接生效。
- 系统为查看历史而创建的 detached preview worktree：按钮为“快照预览”。
- 启动中、重启中、运行中、失败必须有明确状态；不能用“运行中”掩盖后端未启动。
- `frontend-only`、`shared`、`namespace` 继续作为数据能力标签，与实时/快照维度分开显示。

## PR 边界

- `xixi-dev-system`：实时/快照语义、多服务端口、运行证据与项目中心 UI，基于 `cx/branch-preview-dashboard`。
- `canvas-storm`：只提交实时刷新、watch 与适配器相关文件，基于当前开发分支 HEAD。
- `code-quest`：只提交动态端口、数据库命名空间与适配器相关文件，基于当前开发分支 HEAD。
- `gongtu-project`：只提交动态预览与可验证隔离相关文件；题库、课程和现有用户改动不进入该 PR。

## 停止条件

- 需要覆盖、重置或删除原始脏工作副本内容。
- 预览启动可能读写生产数据库、真实账号数据或未授权密钥。
- 动态端口导致无法证明请求属于目标分支，或多个分支写入同一数据库却仍被标记为隔离。
- 项目测试出现与本改动相关的回归，且两轮修复仍不能恢复。
- 远端分支已发生冲突性更新，继续提交会覆盖协作者工作。

## 失败记录要求

允许一次实现尝试失败，但失败不能只留在聊天中。合格记录必须包含：

1. 实际命令与目标分支；
2. 完整日志路径或最小错误摘录；
3. 可重复步骤；
4. 已定位根因；
5. 不会伪装成成功的当前降级状态；
6. 一个可执行、可验收的后续任务。

## Completion Audit

- [ ] 原始需求“正在开发也能实时预览”有真实文件改动触发证据。
- [ ] live 与 snapshot 由工作目录事实推导，不由按钮文案猜测。
- [ ] 三个项目都能从同一项目中心启动；失败项目有合格记录。
- [ ] 端口不冲突；声明隔离的数据路径不相同。
- [ ] 桌面和手机无横向溢出、按钮遮挡和状态歧义。
- [ ] 系统及三个项目的相关测试通过，或失败被限定且可复现。
- [ ] 原始脏工作副本内容未被清理、重置或误提交。
- [ ] 每个发生代码变更的仓库有独立提交、远端分支和 PR 证据。
